# 优化方向与路线图

本文档把当前项目从“可演示 MVP”推进到“可内部试运行系统”的优化方向拆成四层：

1. 产品体验
2. 业务能力
3. 工程架构
4. 上线治理

## 1. 当前判断

项目已经具备物流系统的核心骨架：获客、客户、报价、订单、财务链路都已经打通，Supabase 表结构、RPC、RLS、seed、smoke test 也有基础体系。

下一阶段的重点不是继续堆页面，而是把系统从“能演示”变成“业务人员愿意每天使用”：

- 流程要更像 CRM/TMS/ERP，而不是 demo
- 数据要尽量真实持久化，减少本地草稿
- 高风险规则要迁到后端或 Supabase RPC / Edge Functions
- 订单、报价、财务要补齐列表、详情、状态和审批
- 交付前要有更可靠的测试和部署检查

## 2. 优化优先级总览

| 优先级 | 方向 | 目标 | 建议周期 |
| --- | --- | --- | --- |
| P0 | 路由与系统入口 | 已支持官网和系统模块 URL；下一步补详情页深链 | 1-2 天 |
| P0 | 成本中心管理页 | 已有基础费率维护入口，继续补导入、版本和审批 | 2-4 天 |
| P0 | 订单详情独立化 | 订单从列表进入详情，支持节点、货物、费用、附件 | 3-5 天 |
| P1 | 报价生产化 | 报价版本、审批、PDF/邮件输出、低毛利预警 | 4-7 天 |
| P1 | 财务闭环 | 应收应付、收付款、对账、发票字段和导出 | 5-8 天 |
| P1 | 数据权限收紧 | 角色、RLS、敏感操作后端化 | 3-6 天 |
| P2 | 自动化测试 | 核心业务链路与数据库 smoke 自动化 | 3-5 天 |
| P2 | 运营看板 | 销售漏斗、毛利、订单异常、回款风险 | 3-6 天 |

## 3. 产品体验优化

### 3.1 正式路由

当前已接入 `React Router`，官网层和系统模块可以通过 URL 访问。后续重点是把详情页继续拆出来。

当前已支持：

```text
/
/about
/quote
/system/overview
/system/leads
/system/customers
/system/quotes
/system/cost-center
/system/orders
/system/finance
```

建议继续补：

```text
/system/orders/:id
/system/customers/:id
/system/quotes/:id
/system/rate-sheets/:id
```

收益：

- 刷新后能回到当前模块
- 订单、客户、报价可发链接给同事
- 方便后续做权限守卫和面包屑
- 更适合浏览器前进后退

### 3.2 工作台从“展示型”转为“操作型”

内部系统首页应减少说明性大段文案，增加业务人员每天要看的信息：

- 今日待跟进线索
- 待审批报价
- 待补成本订单
- 异常订单
- 待收款账单
- 低毛利风险

系统界面建议更偏管理后台：

- 列表密度更高
- 筛选更稳定
- 表格列可扫描
- 详情页动作固定在顶部或侧边
- 状态、负责人、下一步动作始终可见

### 3.3 统一交互模式

建议统一每个模块的基本形态：

```text
列表 -> 详情 -> 操作 -> 状态更新 -> 日志
```

适用模块：

- 线索
- 客户
- 报价
- 订单
- 应收
- 应付
- 成本费率

## 4. 业务能力优化

### 4.1 成本中心独立成模块

当前成本中心已被报价使用，但没有独立维护入口。建议新增 `CostCenterWorkspace`。

第一阶段功能：

- 供应商列表
- 费率表列表
- 费率行维护
- 生效日期和失效日期
- 运输方式和货型筛选
- 币种、单位、最低收费

第二阶段功能：

- 费率导入
- 费率审批
- 历史版本
- 客户等级加价规则
- 低毛利拦截

### 4.2 报价中心生产化

当前报价链路可跑通，但还偏 MVP。

建议补齐：

- 报价版本管理
- 报价状态：草稿、待审批、已发送、已接受、已拒绝、已过期
- 低毛利审批
- 正式报价单 PDF
- 报价邮件发送记录
- 报价成本快照锁定
- 销售价和成本价分离

关键原则：

- 报价一旦发送，成本与售价应有快照
- 费率表更新不应影响历史报价
- 低毛利不能只靠前端提示

### 4.3 订单详情独立化

当前订单已支持列表、手工录单和详情数据读取。下一步建议把订单详情变成独立页面。

核心页签：

- 概况
- 基本信息
- 收发通
- 货物
- 运输节点
- 单证附件
- 费用明细
- 异常
- 操作日志

关键动作：

- 状态流转
- 节点更新
- 附件上传
- 异常登记
- 成本确认
- 生成应收应付

### 4.4 财务闭环

当前财务已有应收、应付、付款、成本骨架。生产前要补：

- 应收账期
- 应付账期
- 发票号
- 税率/VAT
- 收款核销
- 付款核销
- 汇率来源
- 对账导出
- 推送用友/外部财务系统状态

建议先把财务流程固定为：

```text
订单费用行 -> 应收/应付 -> 账单 -> 发票 -> 收付款 -> 核销 -> 对账导出
```

### 4.5 获客与销售跟进

线索评分和二次跟进 RPC 已经具备。后续优化可以围绕转化效率：

- 新增线索表单已补，后续做归并和客户时间线
- 线索查重已补基础提醒，后续通过 RPC 做安全合并
- campaign 成效看板已补基础版，后续接广告成本、网站访问和成交收入
- 邮件任务详情
- 邮件发送服务接入
- 销售跟进 SLA
- 线索转客户后的活动时间线

## 5. 工程架构优化

### 5.1 收敛数据写入入口

现在前端可直连 Supabase table 和 RPC。MVP 阶段合理，但生产阶段建议把敏感写操作收敛。

优先迁移到 RPC 或 Edge Functions 的操作：

- 报价核价
- 报价审批
- 报价转订单
- 订单生成应收应付
- 成本确认
- 财务核销
- 用户角色管理

收益：

- 避免关键业务规则暴露在浏览器
- 降低 RLS 策略复杂度
- 更容易写审计日志
- 更容易做幂等和事务

### 5.2 引入测试分层

建议至少补三类测试：

- 构建测试：`npm run build`
- 数据库 smoke：现有 `smoke_test.sql` 和 `npm run check:supabase`
- 核心业务流 E2E：线索 -> 报价 -> 订单 -> 财务

推荐先覆盖这些场景：

- 未配置 Supabase 时页面不崩
- 已登录用户可创建线索/报价/订单
- 报价保存后可转订单
- 订单可生成应收应付
- 低权限角色无法访问不该访问的数据

### 5.3 清理 mock 与真实数据边界

现在 mock/demo 对演示有帮助，但需要边界清楚。

建议：

- demo 数据集中放在 `src/system/mockData.js`
- 页面上明确区分“本地草稿”和“已写入数据库”
- 真实业务模块优先走 Supabase
- 新功能不要继续散落新的局部 mock

### 5.4 统一错误与空状态

每个模块建议统一：

- loading 状态
- empty 状态
- permission denied 状态
- 保存失败状态
- Supabase 未配置状态
- 本地草稿状态

这会显著降低内部试用时的困惑。

## 6. 数据库与权限优化

### 6.1 角色模型

当前角色约定：

- `sales`
- `ops`
- `finance`
- `marketing`
- `manager`
- `admin`

建议补充角色矩阵文档，明确每个角色能做什么：

| 模块 | sales | ops | finance | marketing | manager | admin |
| --- | --- | --- | --- | --- | --- | --- |
| 线索 | 读写本人 | 只读 | 只读 | 读写 | 全部 | 全部 |
| 客户 | 读写本人 | 只读 | 只读 | 只读 | 全部 | 全部 |
| 报价 | 读写本人 | 只读 | 只读成本摘要 | 只读 | 审批 | 全部 |
| 订单 | 只读本人 | 读写 | 只读费用 | 只读 | 全部 | 全部 |
| 财务 | 只读摘要 | 只读成本 | 读写 | 无 | 全部 | 全部 |
| 成本中心 | 只读 | 读写供应商成本 | 只读 | 无 | 审批 | 全部 |

### 6.2 审计日志

建议所有关键动作写日志：

- 报价创建/审批/发送/接受
- 订单创建/状态变更/节点更新
- 成本变更
- 应收应付生成
- 收付款登记
- 用户角色变更

## 7. 建议实施顺序

### 第一阶段：内部试运行基础

目标：让业务人员能稳定用起来。

1. 引入 `react-router`
2. 新增成本中心模块
3. 独立订单详情页
4. 统一 loading / empty / error 状态
5. 补充核心链路 E2E 检查

### 第二阶段：报价与财务生产化

目标：把报价、订单、财务风险收住。

1. 报价版本与审批
2. 报价 PDF/邮件输出
3. 订单费用行完善
4. 应收应付账期和核销
5. 财务导出格式

### 第三阶段：权限、审计和自动化

目标：具备上线治理能力。

1. 角色矩阵
2. RLS 策略复核
3. 敏感写操作后端化
4. 审计日志
5. 部署流水线和回归测试

## 8. 最建议先做的 5 件事

如果只选最有杠杆的优化，建议按这个顺序：

1. 引入路由，让系统模块和详情页有稳定 URL
2. 做成本中心页面，因为报价质量取决于费率维护
3. 拆订单详情页，因为订单是 CRM/TMS/财务的交汇点
4. 报价增加版本、审批、PDF 输出，让销售能真正对外使用
5. 把财务核销和敏感写操作收敛到 RPC / Edge Functions
